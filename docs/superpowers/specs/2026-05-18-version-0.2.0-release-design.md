# Siberia 0.2.0 Release Design

**Goal:** Cut a `0.2.0` release that includes a first-class `siberia --version` command, aligns package metadata and docs with the new version, and publishes the release through the existing GitHub tag workflow.

## Scope

This change should stay narrow:

- bump the version from `0.1.0` to `0.2.0`
- expose `--version` as a top-level CLI flag
- update tests to cover the new flag and the new version string
- update any release-facing documentation that still hard-codes `0.1.0`
- use the existing `v*` GitHub Actions release workflow by pushing tag `v0.2.0`

This change should not redesign the CLI, alter release automation structure, or broaden packaging support.

## Current Context

The repository already contains:

- package metadata in `pyproject.toml`
- an in-code version constant in `src/siberia/cli.py`
- version re-exporting in `src/siberia/__init__.py`
- unit tests in `tests/test_siberia.py`
- a release workflow in `.github/workflows/release.yml` that triggers on pushed tags matching `v*`

The repository also has unrelated untracked files in the worktree. Those should be left untouched unless explicitly requested.

## Proposed Approach

### 1. Version source and metadata alignment

Keep the existing version flow simple: update the current in-package version constant to `0.2.0` and align `pyproject.toml` with the same value.

This preserves the current project structure and avoids introducing a new version-management layer during a release cut.

### 2. CLI version reporting

Add `--version` as a parser-level flag so users can run:

```sh
siberia --version
```

Expected behavior:

- print the semantic version only
- exit successfully without requiring a subcommand
- avoid changing existing subcommand parsing behavior

### 3. Documentation alignment

Update any user-facing release/install examples that still embed `v0.1.0` or `0.1.0` artifact names so the README matches the actual release being cut.

### 4. Release execution

Do not modify `.github/workflows/release.yml` for this release. Use the existing process:

1. commit the release changes
2. push the branch
3. create and push git tag `v0.2.0`
4. let GitHub Actions build artifacts and create the GitHub Release

## Files Expected To Change

- Modify: `src/siberia/cli.py`
- Modify: `pyproject.toml`
- Modify: `tests/test_siberia.py`
- Modify: `README.md`
- Keep as-is: `src/siberia/__init__.py` unless import/export behavior needs a trivial alignment edit
- Keep as-is: `.github/workflows/release.yml`

## Testing Strategy

Use TDD for the new CLI behavior:

1. add a unit test for `main(["--version"])`
2. run that test and confirm it fails for the expected reason
3. implement the minimal parser change
4. rerun the targeted test and confirm it passes
5. run the full unit suite
6. run `python -m build`

Verification should also include checking the final git diff before commit and verifying that the pushed `v0.2.0` tag starts the existing release workflow.

## Risks and Constraints

- There are existing untracked files in the repository; they should not be accidentally staged unless they are intentionally part of the release.
- Documentation must not overstate Cargo support. Current behavior includes `Cargo.lock` auditing in `siberia check`, but not broader Cargo-native install policy management.
- The `--version` implementation should remain top-level and not require any environment-specific setup.

## Success Criteria

The release is successful when all of the following are true:

- `siberia --version` prints `0.2.0`
- unit tests pass
- `python -m build` succeeds
- the intended files are committed and pushed
- tag `v0.2.0` is pushed
- GitHub Actions creates the `0.2.0` release artifacts from that tag
